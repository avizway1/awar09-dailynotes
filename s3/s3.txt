

S3 : Simple Storage Service : 

--> S3 is an Object based storage service
--> We can store flat files. We can store anything, Everything is an object. (.exe, .pdf, .docx, .abc)
--> We need to create a bucket.

bucket = directory with unique namespace

Global Namespace : Bucket name should be unique across the globe
Account Regional Namespace : name should be unique within our account, its automatically add suffix.

Bucket Name:
min : 3 char
max: 63 char
SHould not start with .
should not end with .
no ..
should not resemble IP address format (192.168.100.1)

--> we have unlimited storage in s3 platform.

Min obj size : 0 bytes
Max obj size : 50 TB (earlier its 5 TB)

Standard URL:
https://s3.<region-code>.amazonaws.com/<bucket-name>/<object-name>

Virtual Path (We should not have . in the bucket name): 
https://<bucket-name>.s3.<region-code>.amazonaws.com/objectname
https://<bucket-name>.s3.amazonaws.com/objectname

S3 Pricing : 
--> How many API calls happening on our data.?  (PUT / GET / UPDATE)
--> How much data is traffering.?
--> How much data you are storing in an s3 bucket. (gb/tb / days)
--> What storage class you are using (standard/ IA / Glacier / IT)
--> Depends on Storage class, How much data you are retriving.? 

S3 Storage Classes : 
--> How frequently you are accessing data.?
--> Do you need data without delays.? or is it okay to wait for sometime to retrive the data.?

S3 Standard : frequently accessed data (more than once in a month) with milliseconds latency/access.
--> We can access data without any delays.
--> Data spans across multiple AZs (>= 3 AZs)
--> 99.99% Availability and 99.999999999% (11 9's) durability

Standard-IA (Infrequently Accessed) : 
--> We can access data without any delays.
--> Store here if you access data infreq (once in a month) with missiseconds latency.
--> Data spans across multiple AZs (>= 3 AZs)
--> 99.99% Availability and 99.999999999% (11 9's) durability

OneZone-IA : 
--> We can access data without any delays.
--> As data stores only in 1 AZ, Recommended to store easily reprodusable/recretable data.
--> Store here if you access data infreq (once in a month) with missiseconds latency.
--> Data stores only in 1 AZ
--> 99.95% Availability and 99.999999999% (11 9's) durability

Glacier Instant Retrival : 
--> Long loved archieve data, but accessed once a querter with instant retrival in milliseconds.

Glacier Flexible Retrival (Formerly Glacier):
--> long lived archieval data, accessed once a year with retival time between 2 - 5 hrs.
--> We cannot access data immediately.

Glacier Deep Retrival:
--> long lived archieval data, accessed less than once in a year with retival time between 12-24 hrs.
--> We cannot access data immediately.

Intelligent Tier : 
--> Data with changing or unknown access patterns
--> Its little contlier comapred to other storage classes.

Glacier Storage : Perform data retrival.. 
Bulk Retrival : Takes 5-12 hrs
Standard Retrival : Takes approx 3-5 Hrs
Expdited Retrival : Takes 1-5 min (Object size should not exceed 250 MB)

Object lock : WORM Model (Write Once and Read Many) helps you to prevent object deletion. 

We have 2 modes.. 
1. Governance Mode : User who have valid IAM permissions can overerite or delete protected objects, during retention period.
2. Compliance Mode : No user can overwrite or delete protected object versions during the retention period (Including the root user.)

In-Transit Encryption / At flight encryption :  When data is in trasit state, we cna encrypt the data. Its aws responsibiloity, aws adds ssl/tls certificate for web urls, so data will be encrypted automatically.

Client side encryption / CSE : We can encypt data before uploading it to s3 platform, customer/we are responsible for this encryption method/type/decryption.

Server Side encryption / SSE : When data is in S3 platform, We can apply encryption methods on our data/s3 bucket.


1). Who Manages/generates the key material. 
2). Who can decrypt it.


SSE-S3 : Default encryption method for s3 data..
--> S3 generates and manages the key material
--> Whoeverhas has access to s3 platform, they can decrypt the data automatically.. No addl permisisons required to decrypt the data..
--> Suitable for public sharable data.. 
--> No Direct access to view/manage key material.. 

---

KMS : key management service : THis service helps to create, store, rotate anc control encryption keys. 

SSE-KMS  DMK (Data Management key / Default Master key / Data Master key): 
--> KMS generates and manages the key material. 
--> Whoeverhas has access to s3 platform, they can decrypt the data automatically.. No addl permisisons required to decrypt the data..
--> No Direct access to view/manage key material.. 
--> We cannot make objects public. 
--> This key generates automatically whenever we start using the service. 

---

SSE-KMS CMK (Customer Managed Key).. 

--> Customer has to create the KMS Key.. KMS manages the key material. 
--> Only s3 service access is not sufficient, User should have permissions on Key Usage as well, along with S3 service access..
--> No Direct access to view/manage key material.. 
--> We cannot make objects public. 
--> We can rotate the key material., We generally configure key rotation for every 365 days.

Step 1 : Choose the key type.

Symmetric Key : A single key will be used for encryption and decryption purposes : S3 supports only Symmetric key.. 1 Shared key.. Extremly fast.. Bulk data and large files.. 

Asymetric key : Just like connecting to ec2 instance using keypair (ec2 instance will have public key, we will have private key).. A public and private key used for encryption and decryption data. 


Step 2 : Provide the alias name and description.

Step 3 : Choose the key administrator.		: Avinash_T

Step 5 : Review and create the key.
