If any changes happened outside of the terraform, We can identify, We call it as "drift detection"..

1. Just Detect			: terraform plan -refresh-only
2. Accept the drift		: terraform apply -refresh-only (when someone made valid change - update statefile)
3. Revert the drift		: terraform apply				(our code is truth - undo the manual change)

We always store the "tfstate file" inside an "S3 bucket".
1. Enable versioning on s3 bucket.
2. Enable s3 bucket object lock with "compliance mode", So no one can delete the file, Even if the root user try to delete, if fails till the retention day, we configure.


terraform {
  backend "s3" {
    bucket = "aviz.tf.state.myproject"
	key = "projectx/terraform.tfstate"
	region = "ap-south-1"
	use_lockfile = true
  }
}


*Rare cases, we do this
terraform force-unlock <lockid>

modules:

A Module is a reusable, packaged terraform configuration.

root module : the main directory where you run terraform apply.
child module : A module called by the root module.
public module : a module from the terraform registry.

provider block always should be in root module only. 
run "terraform init" whever you add a new module.
each child and root module require "variables.tf" file, even if you have not setting default values.
