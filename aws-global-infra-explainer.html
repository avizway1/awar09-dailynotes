Core Concepts
The Three Layers of AWS Infrastructure
🌍
Region

A Region is a physical geographic area in the world that contains multiple, isolated data centers (AZs). Think of it as an AWS "city" — a complete, independent cluster of infrastructure.

  • 33+ Regions worldwide (2025)
  • Fully isolated from other Regions
  • Each has 3–6 Availability Zones
  • Example: us-east-1 (N. Virginia)
  • You choose the Region for compliance
🏢
Availability Zone

An AZ is one or more physical data centers within a Region, each with independent power, cooling, and networking. They are connected via high-speed, low-latency links.

  • 105+ AZs globally
  • Isolated from failures in other AZs
  • ~1–2ms latency between AZs
  • Example: us-east-1a, us-east-1b
  • Deploy across AZs for HA
⚡
PoP / Edge Location

Points of Presence (PoPs) are mini AWS sites closer to end users. Used by CloudFront CDN to cache and serve content at the network edge — drastically reducing latency.

  • 600+ PoPs across 90+ cities
  • Used by CloudFront, Route 53
  • Caches content near the user
  • No compute — cache & routing only
  • India: Mumbai, Chennai, Delhi…

Cache HIT flow: Route53 uses DNS to direct you to the nearest CloudFront PoP (Mumbai). The PoP has the file cached from a previous request — so it responds instantly without touching S3 in Virginia. Latency drops from 200ms → ~10ms. 🚀

Cache MISS flow (first request): Mumbai PoP doesn't have the file → fetches from S3 in N.Virginia via AWS's private backbone → caches it → returns to user. All subsequent requests are served from Mumbai.
SCENARIO 03
🗑 Cache Invalidation — Clearing PoP Locations
When to use it · How it works · AWS CLI command
1

Update S3

Upload new version of file to S3 bucket in us-east-1

2

Old Cache Remains

Mumbai, Singapore, Frankfurt PoPs still serve old cached version

3

Create Invalidation

Trigger CloudFront invalidation for the path you updated

4

PoPs Cleared

All edge locations delete cached copy of that file path

5

Next Request

Cache miss → PoP fetches fresh version from S3 → caches new file

// AWS CLI — Create Invalidation
# Invalidate a specific file path aws cloudfront create-invalidation \ --distribution-id E1A2B3C4D5E6F7 \ --paths "/images/logo.png" # Invalidate all files (wildcard) aws cloudfront create-invalidation \ --distribution-id E1A2B3C4D5E6F7 \ --paths "/*" # Check invalidation status aws cloudfront list-invalidations \ --distribution-id E1A2B3C4D5E6F7
💡 When to Invalidate: When you upload a new version of a file to S3 but CloudFront is still serving the old cached version. Without invalidation, users see stale content until TTL expires (default: 24hrs).

⚠️ Cost Tip: First 1,000 invalidation paths per month are free. After that, $0.005 per path. Use versioned file names (logo-v2.png) to avoid invalidations entirely — a better production strategy!
